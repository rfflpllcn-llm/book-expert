# Knowledge Base Guide

How to structure and write the knowledge files that power a book's expert agent.

## Architecture

The knowledge base uses a three-tier architecture optimized for prompt caching with the Anthropic API:

```
knowledge/
├── tier_1/          # ~10K tokens — cached in system prompt, loaded every query
├── tier_2/          # ~200K tokens — loaded on demand per query
├── tier_3/          # essay routing metadata — always loaded
│   └── _index.yaml  # structured YAML: essay summaries + section descriptions
└── answers/         # grows over time — cached full answers
```

**Why three tiers?** Tier 1 stays in the Anthropic API's prompt cache (5-minute TTL), so it's nearly free after the first query. Tier 2 is loaded only when the query matches an arc, keeping per-query costs proportional to relevance. Tier 3 adds critical depth without inflating every response.

## Tier 1 — Always-Loaded Context

These files form the agent's "mental map" of the book. They are concatenated and placed in the system prompt with `cache_control: ephemeral`.

### Writing guidelines

- **Total budget: ~10K tokens.** Going over increases cache-miss cost on every cold start.
- Write for an LLM reader: dense, structured, no filler.
- Use consistent formatting so the agent can parse references.

### File-by-file specification

#### `00_index.md`

The master index. Contains:
- File map (what's in each tier)
- Arc reference table (arc ID, description, line range)
- Scene count and structure overview
- Any numbering conventions (SC_xxxxx, Lnnn)

```markdown
# Index — [Book Title]

## File Map
- `tier_1/` — synopsis, characters, themes, style, context, cache
- `tier_2/` — 15 arc files with scene-level detail
- `tier_3/` — critical commentaries

## Arc Reference Table
| Arc ID | Description | Lines |
|--------|-------------|-------|
| 02_01_tostes | Charles's first marriage, arrival in Tostes | 1–500 |
| 02_02_vaubyessard | The ball at La Vaubyessard | 500–900 |

## Conventions
- Scene IDs: SC_00001 through SC_02630
- Line refs: L1 through L20425 (map to FR1–FR20425 in CSV)
```

#### `01_synopsis.md`

A complete plot summary organized by arc. Each arc gets 2-4 sentences. The agent uses this to orient itself before loading tier 2 detail.

#### `03_characters.md`

Every named character with:
- Which arcs they appear in
- Key traits and relationships
- Approximate scene count

Format as a structured list, not prose:

```markdown
## Emma Bovary
- **Arcs**: 02_01_tostes, 02_02_vaubyessard, 02_03_yonville, ...
- **Role**: protagonist
- **Traits**: romantic idealism, dissatisfaction, self-destruction
- **Relations**: wife of Charles, lovers Rodolphe and Léon
- **Scenes**: ~450
```

#### `04_themes.md`

Each theme gets:
- A one-paragraph description
- Which arcs it's most present in
- 2-3 key scene references

#### `05_style.md`

Language, narrative technique, recurring metaphors, register. Specific to the author — argot for Céline, style indirect libre for Flaubert, etc.

#### `06_context.md`

Author biography (relevant to the work), publication history, censorship/controversy, literary influence, critical reception.

#### `08_qa_cache.md`

Starts nearly empty. Grows as the agent answers questions. Each entry is appended by `lib/save_qa.py`:

```markdown
# Risposte validate — Q&A Cache

---

## Q: Who is Robinson?
**SC di riferimento**: SC_00293, SC_00367
**Risposta**: Robinson is Bardamu's dark double, appearing across four arcs...
**Risposta completa**: answers/2026-03-22_chi-e-robinson.md
**Data**: 2026-03-22
```

Do not edit this file manually unless correcting errors.

## Tier 2 — On-Demand Arc Files

One markdown file per narrative arc. The filename must match the arc key in `book.yaml` exactly:

```
book.yaml arcs key:  02_03_yonville
tier_2 filename:     02_03_yonville.md
```

### Content structure

Each file contains scene-by-scene summaries for that arc. Recommended format:

```markdown
# Arc 3 — Yonville

## SC_00150 — Arrival at Yonville (L500–L520)
Emma and Charles arrive at Yonville-l'Abbaye. First meeting with Homais
at the Lion d'Or. Emma notices Léon across the table.
- **Characters**: Emma, Charles, Homais, Léon
- **Themes**: provincial life, first encounter
- **Tone**: ironic description of Yonville's mediocrity

## SC_00151 — The pharmacy (L521–L540)
...
```

### Writing guidelines

- Include scene IDs (`SC_xxxxx`) and line ranges (`Lstart–Lend`) for every scene
- List characters, themes, and tone per scene — the agent uses these for cross-referencing
- Write summaries in the language specified in the tier_1 files (typically Italian for this project)
- Budget: each arc file can be 5K–15K tokens; they're only loaded when matched

## Tier 3 — Critical Essays

Optional but valuable. Contains secondary/critical sources that enrich answers.

### `_index.yaml` — Essay Routing Metadata

Structured YAML file generated by `lib.aggregate_essay_index`. Contains per-essay
entries with: author, work, year, summary, stance, arcs, themes, characters, and
chapter-level section descriptions.

The agent reads these summaries to decide which essay arcs to load via `cite_essay`.
No keyword matching — the LLM routes semantically.

### Essay knowledge bases

Each essay lives in `data/essays/<slug>/` with its own tier_1, tier_2, filtered JSONL,
and YAML descriptor. These are produced by the same preprocessing pipeline (stages 0-4)
used for the book itself.

## Answers — Cached Full Responses

The `knowledge/answers/` directory is auto-populated by the agent via `lib/save_qa.py`. Each file is a complete answer preserved as markdown:

```
answers/2026-03-22_chi-e-robinson.md
answers/2026-03-27_introduction-to-voyage.md
```

These are linked from `08_qa_cache.md` so the agent can retrieve a full previous answer without re-deriving it.

## Tips for Building a Knowledge Base from Scratch

1. **Start with tier_1.** A good synopsis + character list + theme overview is enough to get the agent working. Tier 2 detail can be added arc by arc.

2. **Use Claude to help.** Feed the source text to Claude and ask it to generate scene summaries, character profiles, and theme analysis. Then review and edit.

3. **Test incrementally.** After writing each tier_1 file, run the agent and see what it gets right and wrong. Add tier_2 arcs for the areas where it struggles.

4. **The QA cache is self-improving.** Every answered question gets cached. Over time, common questions are answered instantly from cache rather than re-derived from the knowledge base.

5. **Commentaries can wait.** Tier 3 is the last priority. The agent works well with just tier_1 and tier_2. Add commentaries when you want richer critical analysis.
